## 1. 产品定位重述：什么是一网统飞

从政务协同与低空监管视角看，“一网统飞”不是某个独立模块，也不是单纯的无人机控制台，而是整个系统的顶层运行理念，要求平台同时覆盖以下 4 类能力：

1. 飞行前：空域资源治理、飞行计划、申请审批、合规校验、放飞授权
2. 飞行中：实时监管、态势感知、视频与遥测、越界与异常告警、应急接管
3. 飞行后：事件闭环、取证留痕、任务复盘、影像归档、监管报表
4. 平台层：多单位协同、权限分层、审计追踪、AI 辅助识别与辅助决策

如果只具备“设备管理 + 调度展示 + 事件演示 + AI 演示”，产品更像“无人设施控制后台”；
只有把“空域治理 + 审批协同 + 放飞监管 + 合规留痕”立起来，才更接近“一网统飞”。

## 2. 一网统飞目标能力地图

### 2.1 一级能力域

| 能力域 | 目标说明 |
|--------|----------|
| 空域资源中心 | 管理禁飞区、限飞区、临时空域、空域走廊、高度层与时间窗口，是统飞能力的支撑子域之一 |
| 飞行协同中心 | 统一承接飞行计划、申请、审批、会签、放飞授权、撤销与补正 |
| 运行监管中心 | 实时地图、视频流、遥测、机巢状态、任务态势、越界与异常告警 |
| 任务执行中心 | 任务编排、飞行执行、返航中断、资源调度、应急处置 |
| 事件处置中心 | 告警转事件、复核、工单、取证、闭环报告 |
| AI 智能中心 | 数据源、媒体库、任务编排、在线检测、检测历史、模型资产 |
| 平台治理中心 | 多单位、角色权限、审计留痕、监管报表、系统参数 |

### 2.2 关键业务主链

建议后续产品叙事统一围绕这 5 条主链展开：

1. 空域治理链：空域资源配置 -> 生效发布 -> 飞行冲突检测 -> 监管联动
2. 统飞审批链：计划上报 -> 飞行申请 -> 审批会签 -> 放飞授权 -> 授权回收
3. 运行监管链：飞行中监控 -> 实时遥测/视频 -> 越界/失联/低电告警 -> 应急接管
4. 事件闭环链：AI / 人工发现 -> 事件复核 -> 工单处置 -> 报告归档
5. 数据复盘链：飞行轨迹 / 视频 / 影像 / 事件 / 审批记录 -> 复盘分析 -> 监管报表

## 3. 当前项目的能力现状

结合 [DEV_GUIDE.md](../../DEV_GUIDE.md)、[DEV_PLAN.md](../../DEV_PLAN.md)、[ROUTES.md](../ROUTES.md) 与现有路由，当前仓库已经具备以下基础：

### 3.1 已具备的基础骨架

- 指挥中心 / 数据驾驶舱一体化入口：`/situation-board`
- 调度中心、设备中心、飞控中心、事件中心、AI 中心的完整菜单框架
- 机巢一键起飞、机巢监控、虚拟座舱等演示型设备控制链路
- 事件复核、工单、报告的前端闭环演示链路
- AI 数据源、媒体库、任务、规则、检测、历史的演示级主链
- 禁飞区、底图、图层、地理资源等基础台账页面

### 3.2 当前更像什么

当前产品更像：

- 一个“智慧无人设施控制后台”
- 一个“多模块演示型低空业务控制台”
- 一个“设备控制 + 事件闭环 + AI 演示”的综合前台

但还不够像：

- 一个“空域治理平台”
- 一个“统飞审批协同平台”
- 一个“运行监管与合规留痕平台”

## 4. 关键能力差距清单

以下差距按“离一网统飞产品定位有多关键”排序。

### 4.1 P0：最核心的产品能力差距

#### G1. 缺少“统飞审批闭环”

当前存在：

- 航线申请、计划、需求等页面

当前缺少：

- 飞行申请单状态机
- 审批 / 驳回 / 补正 / 会签
- 放飞授权、撤销授权、授权有效期
- 审批留痕与审计视图
- 计划、申请、审批、执行、归档的一致编号体系

这意味着当前还没有“一网统飞”最核心的业务骨架。

#### G2. 缺少“空域资源中心”

当前存在：

- 禁飞区管理
- 底图、图层、资源台账

当前缺少：

- 空域网格 / 走廊 / 空域分层
- 临时空域、生效窗口、时段占用
- 飞行申请与空域资源冲突检测
- 空域资源视角的全局地图
- 统一的空域资源状态看板

当前的禁飞区更多是配置项，不是空域治理中心。

#### G3. 缺少“飞前合规检查网关”

当前存在：

- 系统参数中的起飞前空域检查等配置项
- 飞手、设备、禁飞区等分散页面

当前缺少：

- 统一的起飞前校验流程
- 飞手资质、设备健康、空域合法性、天气、风险等级的汇总判断
- 放飞前 checklist
- 校验结果留档
- 不通过时的阻断与人工放行机制

#### G4. 缺少“实时监管接真”

当前存在：

- 指挥中心态势页
- 机巢监控页
- 座舱页

当前缺少：

- 真地图
- 真视频流
- 真遥测数据
- 越界 / 失联 / 偏航 / 低电量 / 返航异常的实时事件流
- 监管视角的大屏与处置席位

当前监管能力仍以 mock / 占位为主，无法支撑“统飞平台”的可信监管叙事。

### 4.2 P1：高价值但可在第二步补齐

#### G5. 缺少“飞行后复盘与监管报表”

当前事件闭环报告相对成熟，但统飞平台更需要：

- 飞行架次、时长、空域使用率
- 申请通过率 / 驳回率 / 驳回原因
- 部门、区域、机队、飞手维度统计
- 告警类型、违规类型、处置效率
- 周报、月报、专项监管报告

#### G6. 缺少“多主体协同治理”

当前更像单后台，仍缺：

- 区县 / 部门 / 单位 / 运营方分层视图
- 多单位隔离
- 监管角色、审批角色、执行角色、值班角色的能力边界
- 横向协同入口与统一工作台

#### G7. 缺少“统一任务编排中枢”

当前任务能力分散在：

- 调度中心
- 飞控中心
- AI 分析任务
- 机巢起飞演示链

仍缺：

- 任务总中心
- 任务类型统一模型
- 人、机、场、空域、AI 策略统一编排
- 任务阶段状态机与回写

### 4.3 P2：增强平台壁垒的能力

#### G8. AI 与统飞监管的深度融合不够

当前 AI 主链已经有基础，但更偏“检测演示”。后续可增强：

- AI 风险预判
- AI 辅助审批建议
- AI 异常轨迹识别
- AI 对多路视频 / 多机协同的监管辅助
- 模型效果与监管效果联动评估

#### G9. 三维空域与时空复盘不足

当前仓库已保留三维扩展位，但未形成平台能力：

- 三维空域展示
- 飞行轨迹立体回放
- 高度层、禁飞层、任务层叠加
- 事件 / 轨迹 / 视频 / 审批记录一体复盘

## 6. 下一阶段建议路线图

### 6.1 Wave 1：先把“一网统飞”主链立起来

目标：让产品从“控制台”变成“统飞平台”。

优先做：

1. 飞行计划 -> 飞行申请 -> 审批 -> 放飞授权 -> 执行 -> 归档 闭环
2. 空域资源中心基础版
3. 飞前合规校验流程

建议新增页面：

- `/airspace/resource`
- `/airspace/apply`
- `/airspace/approval`
- `/airspace/authorization`

建议先做前端演示闭环，再接后端状态机。

### 6.2 Wave 2：补“监管可信度”

目标：让统飞不是纸面流程，而是真实运行平台。

优先做：

1. 指挥中心接真地图
2. 机巢监控 / 指挥中心 / 座舱接真视频流
3. 遥测、控制权、任务阶段统一状态机
4. 越界 / 失联 / 低电 / 偏航告警事件流

### 6.3 Wave 3：补“飞后复盘与监管报表”

优先做：

1. 飞行日志与轨迹归档
2. 飞行复盘页
3. 监管统计报表页
4. 审批与执行留痕联查

### 6.4 Wave 4：补“多主体治理与 AI 增强”

优先做：

1. 多单位 / 多角色 / 多部门权限视图
2. AI 辅助审批、辅助监管、异常识别
3. 模型效果与监管结果联动分析

## 8. 对当前项目的最终判断

当前项目已经具备“智慧无人设施控制后台”的良好基础，尤其在以下方向已有明显积累：

- 设备控制演示
- 事件闭环演示
- AI 检测主链演示
- 指挥态势与调度展示

但如果要对外以“空域应用平台（一网统飞）”来定位，当前最需要补强的不是再多做几个独立页面，而是尽快把以下 4 条产品主线立起来：

1. 空域治理
2. 审批协同
3. 放飞监管
4. 合规留痕

只有这 4 条主线成立，系统才能从“控制后台”真正升级为“统飞平台”。
